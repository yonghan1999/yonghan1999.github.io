# Repository Guidelines

## 项目结构与目录约定

本仓库是一个 Hexo 静态博客。全局配置在 `_config.yml`，正式文章位于 `source/_posts/`，草稿文章位于 `draft/`，预发布文章位于 `pre/`，文章模板位于 `scaffolds/`，静态根文件如 `source/CNAME` 也放在 `source/` 下。主题目录在 `themes/`，当前启用主题为 `keep`，并通过 git submodule 管理；`maupassant` 目录存在但当前未启用。构建产物输出到 `pub/`，不要手动编辑该目录内容。

## 文章书写规范

新文章使用 Markdown，文件名建议采用日期前缀，例如 `source/_posts/2026-07-08-example-title.md`。Front matter 保持简洁，示例：

```yaml
---
layout: post
title: 示例标题
date: 2026-07-08
categories: 技术
tags: Hexo 博客
---
```

文章以中文技术笔记风格为主，要求问题背景、操作步骤、命令示例、结论清晰。表达上优先通俗易懂、生动形象，复杂概念要配合类比、流程图、结构图或示意图说明，避免只堆概念。叙述尽量使用第一人称视角，保持同学间互相交流、共同梳理问题的口吻。可以分享“我一开始怎么理解”“我在哪个地方卡住”“目前我是这样梳理的”，避免“我来教你”“你先记住”“不要急着”等居高临下或替读者安排学习过程的表达。引用外部工具、协议或平台行为时，如果版本或日期会影响准确性，需要写明上下文。

## 文章流转约定

当用户要求编写一篇文章或一个系列文章时，必须先在 `draft/` 目录中创建和修改 Markdown 文件。`draft/` 是唯一允许反复迭代的工作区。

只有得到用户明确同意后，才可以将 `draft/` 中的文章复制或移动到 `pre/` 目录。后续如果仍需修改文章，必须继续修改 `draft/` 中的对应文件；再次得到用户明确同意后，再用 `draft/` 文件替换 `pre/` 中的对应文件。

只有当用户明确要求发布时，才可以将文章从 `pre/` 发布到 `source/_posts/`。文章一旦进入 `source/_posts/` 即视为正式发布版本，不再允许修改对应正式文件。发布完成后，必须删除 `draft/` 和 `pre/` 中对应的文章文件，避免多版本残留。

## 验证约定

文章编写、预发布和发布流程不需要执行 `npm install`、`npm run build`、`npm run server` 等 npm 编译或预览验证命令。除非用户明确要求，不主动进行 npm 相关验证。

## Git 提交约定

完成草稿、发布到 `pre/`、正式发布到 `source/_posts/` 这三个节点都需要进行 git commit。每次提交只包含当前阶段相关文件，避免混入其他草稿、配置或未确认内容。提交信息保持简短直接，例如 `add k8s draft`、`pre return post`、`publish return post`。

## 分类与标签约定

优先复用已有分类和标签，例如 `技术`、`工具`、`生活`、`软考`、`Java`、`Git`、`Hexo`、`WebRTC`。避免创建仅大小写、空格或中英文写法不同的重复标签。标签数量保持克制，只标记文章的核心主题。

## 代码与配置风格

配置文件保持现有 YAML 风格，嵌套缩进使用两个空格。代码块必须标注语言类型，例如 `bash`、`yaml`、`java`。不要为了单篇文章添加未使用依赖、全局配置或主题定制；遵循 KISS、YAGNI，保持改动范围最小。
